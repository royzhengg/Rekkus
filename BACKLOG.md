# Rekkus — Execution Backlog

`BACKLOG.md` is execution truth. It preserves shipped history, sequences work top-to-bottom, and turns strategy into actionable tasks for humans and AI agents.

Strategic truth: [PRODUCT.md](PRODUCT.md)  
Agent behavior: [AGENTS.md](AGENTS.md)  
AI constraints: [AI_RULES.md](AI_RULES.md)  
Architecture truth: [docs/architecture/ARCHITECTURE.md](docs/architecture/ARCHITECTURE.md)

Status: `[ ]` not started · `[~]` in progress · `[x]` shipped

Lifecycle model: backlog rows use checkbox status for execution state and Implementation Type for delivery shape. Use `[ ]` for not started, `[~]` for actively scoped or partially shipped work, and `[x]` only when implementation evidence exists and related docs/checks are updated. Preserve completed rows as shipped history; reopen with `[~]` when evidence exists but a material gap remains. Docs-only shipping is valid only when the row itself is strategy, owner-index, policy, ADR/template, or intentionally deferred external/legal work; rows that claim validation, automation, runtime behavior, data model support, privacy workflow, media safety, jobs, auditability, or release gating need actual implementation evidence.

ID allocation: reserve new backlog IDs in [backlog-counter.md](backlog-counter.md) before adding rows. Immediately increment the counter after reservation. Never guess, reuse, or renumber IDs.

Known duplicate policy: do not rewrite history automatically. If duplicate IDs appear, document the conflict and plan a safe migration before changing existing rows.

## Backlog Column Guide

Backlog section tables use this schema: `| Status | Priority | ID | Item | Problem | Implementations | Implementation Type |`

**Reduced from 11 → 7 columns.** Four columns (Why It Matters, Dependencies, Burden, Suggested AI Command) were removed and their content absorbed into Problem so rows fit in agent context windows without truncation. When adding a row, put everything an agent needs into Problem: the problem statement, why it matters, dependencies (`Depends on: ...`), burden if not Low (`Burden: Medium/High`), and the AI instruction (`Do: ...`).

| Column              | What It Must Contain                                                                                                                                                                                                                                                                                                                 |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Status              | `[ ]` not started, `[~]` actively scoped or partially shipped, `[x]` shipped with implementation evidence plus docs/checks, or `Deprioritized` for intentionally parked work. Only set `[~]` when Implementations contains partial shipped evidence — never `[~]` with “Not implemented yet.”                                         |
| Priority            | Current execution importance, not long-term strategic ambition.                                                                                                                                                                                                                                                                      |
| ID                  | Stable backlog reference; never reuse or renumber outside the ordered backlog sequence.                                                                                                                                                                                                                                              |
| Item                | Specific deliverable: name the actual system, guardrail, workflow, UI, migration, automation, or doc owner. Avoid vague labels like “setup” or “plan” unless the deliverable really is a plan.                                                                                                                                       |
| Problem             | All context an agent needs in one cell: crisp problem statement, why it matters (product/compliance/cost/safety reason), `Depends on: ...` for prerequisites, `Burden: Medium/High` if not Low, `Do: ...` AI instruction. Include `Compliance Impact:` for risky privacy, provider, security, media, analytics, ranking, or payment changes. |
| Implementations     | Shipped evidence with paths/checks, or the exact reason the item is not implemented.                                                                                                                                                                                                                                                 |
| Implementation Type | One of `runtime-feature`, `migration`, `guardrail`, `automation`, `ops-workflow`, `docs`, `restructure`, `audit`, `policy`, or `none`.                                                                                                                                                                                               |

## Execution Principles

- Prioritize operational visibility → trust and safety → cost efficiency → release safety → maintainability → product utility → retention → growth loops → advanced discovery → advanced AI systems.
- Build foundational leverage before feature scale: observability, security, caching, release safety, moderation, and data integrity first.
- Keep Google as enrichment, not core serving infrastructure.
- Prefer deterministic systems before AI.
- Preserve completed items for institutional memory.
- Treat the master plan as high-level operating truth and this file as execution truth.
