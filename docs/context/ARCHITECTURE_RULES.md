# Architecture Rules

These rules apply to every feature. They do not change per-task.

1. **Never query Supabase from `app/`, `features/`, `lib/hooks/`, or `lib/contexts/`.** All Supabase access lives in `lib/services/`. Enforced by ESLint rule `check:architecture`.

2. **Never edit `types/database.ts`.** It is generated by `supabase gen types typescript`. Re-run generation if the schema changes.

3. **Never edit `supabase/schema.sql`.** It is a generated artifact assembled from `supabase/schema/` source files. Edit the source SQL files instead.

4. **Search indexes and `place_stats` are derived — never authoritative.** Rebuild from source tables when stale. Never patch them directly to fix a data issue.

5. **Use existing service functions before creating new ones.** Check `lib/services/places/queries.ts`, `lib/services/users/queries.ts`, etc. Duplication creates diverging behaviour.

6. **One domain owns one table family; cross-domain reads go through the owning service.** Do not import the places service from the messaging service or vice versa. Cross-domain data is assembled at the feature or hook layer.

7. **`Database['public']['Tables']` must not appear outside `lib/types/` or `types/`.** Features and hooks use the type aliases from `lib/types/` or the domain types from `types/domain.ts`.

8. **Notifications are delivery-only.** Never drive business logic from `notification_deliveries` rows. Source of truth is always the originating domain table.

9. **`follow_requests` and `follows` are separate tables with separate lifecycles.** An approved follow request creates a row in `follows`; the request row is terminal but not deleted.

10. **Never cast to a branded ID directly.** `"abc" as PlaceId` is forbidden. Use the constructor functions in `lib/types/branded.ts`: `asPlaceId("abc")`, `asUserId("abc")`, etc.
